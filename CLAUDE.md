# CLAUDE.md

## 0. このファイルの目的

このファイルは、Claude Code が本プロジェクトで作業する際に必ず参照する開発ルールである。

本プロジェクトは、単なる入力フォームやLPではなく、以下が相互に連動している。

- 入力フォーム
- 診断ロジック
- 結果表示
- 課金導線
- メール生成
- 法律・行政・不動産実務上の表現リスク
- ユーザーが「自分の案件をどう確認すべきか」を理解する体験

そのため、局所的な修正だけで判断してはいけない。

修正前後には、必ず全体の接続とサービス思想への影響を確認すること。

特に、本プロジェクトは旧フローから新フローへの移行途中である。  
表示されていないファイル、古い名前のファイル、現在 import されていない処理であっても、課金・メール生成・既存API連携・将来移植予定の処理が含まれている可能性がある。

推測で「未使用」「不要」「削除可能」「接続済み」「移植済み」と判断してはならない。

## 1. このサービスの目的と思想

### 目的

賃貸契約における費用について、借主が「確認すべき論点」と「次の行動」を自分で整理できる状態を作ること。

### 診断の2軸

**軸A：説明義務・判断可能性（主軸）**

宅建業者・管理会社・仲介会社等の不動産事業者側が、借主に対して、費用の性質・根拠・選択肢を理解した上で判断できる状態を提供していたかを確認する。

違法かどうかを断定するのではなく、  
「この説明で借主は判断できたか」を問う。  
行政的観点・不動産実務上の観点から、確認対象になり得るかを整理する。

**軸B：法的一般論（副軸・できる範囲で維持する）**

宅建業法・消費者契約法・国交省ガイドライン・賃貸借実務等の一般的な制度情報を、「一般的には〜とされています」「確認材料になり得ます」という形で提供する。

個別事案への法的評価・違法性の断定は行わない。

この2軸は同時に走らせる。どちらか一方に寄せない。

- 法律的観点を消して、単なるゆるい相談フォームにしてはならない。
- 逆に法的断定を強めて、違法確定・返金確定サービスにしてもならない。

### 出力の設計思想

表現は一般論・確認促進にとどめる。

ただし、ユーザーの入力内容に即して、  
「この状況では構造上ここが問題になり得る」  
という答えが自然に浮かび上がるように設計する。

ユーザーが「自分の案件が診断されている」と感じる個別性を持たせる。

一般論として説明しながらも、入力内容に即した構造的な答えが見える状態を作る。

> 断定しない。  
> しかし、答えが見える状態を作る。

## 2. 費目の3分類（重要）

診断ロジック・質問設計・結果表示は、この3分類に基づいている。

### 1. 外せる費目（任意性が核心）

対象例：

- 消毒代
- 24時間サポート
- 書類作成費・事務手数料

基本方針：

- 外せる可能性があることを先に伝える
- 外せないと言われた場合は、実態・根拠・説明内容を評価する
- 削除が難しい場合は、フリーレント転換などの交渉余地を提示する

### 2. 実態の検証が必要な費目

対象例：

- 鍵交換代
- クリーニング費

基本方針：

- 任意性の確認
- 実施実態・第三者検証可能性の確認
- 領収書・作業日・実施業者・作業内容・鍵番号等の根拠資料の有無を確認する
- 第三者にも説明できる根拠資料の確認が重要であることを示す

### 3. 制度的な根拠確認が必要な費目

対象例：

- 仲介手数料
- 保証会社費用
- 火災保険料
- 礼金

基本方針：

- 制度上の原則を先に伝える
- 説明義務の確認
- 任意性・代替可能性・利益相反・グループ会社関係の有無を確認する

---

## 3. 仲介手数料の核心論点（特に重要）

仲介手数料では、以下の順序が問題になる。

1. 原則0.5ヶ月分であることの説明があったか
2. その上で1ヶ月分への承諾を求められたか
3. 書面による承諾があったか
4. 貸主からも仲介手数料を受け取っていないか、または合算上限との関係が説明されているか

①なしに②③があっても、承諾の有効性・説明の十分性が問われる余地がある。

この順序をコードが正しく反映しているかを常に確認すること。

## 4. 出口設計（重要）

現行設計では、以下の出口設計を基本とする。

```text
削除を求める（delete）
↓ 断られた場合
フリーレント転換を提案する（free_rent）
↓ 断られた場合
行政ほのめかし・根拠確認（admin_check / confirm）
↓ 条件が変わらない場合
他社との見積もり比較を案内する
```

この出口設計を、明示的な指示なしに変更してはならない。

## 5. 禁止する表現・推奨する表現

### 禁止

* 「違法です」
* 「必ず返金されます」
* 「この請求は無効です」
* 「裁判で勝てます」
* 「業者が詐欺をしています」
* 「行政処分されます」
* 「相手は法律違反です」と断定する表現
* 「このままだと必ず損をします」
* 有料購入を急がせるための過度な危機感の演出
* 不安を煽って課金へ誘導する表現

### 推奨

* 「確認が必要な状態です」
* 「説明を求める余地があります」
* 「一般的には注意が必要な請求です」
* 「判断するには根拠資料の確認が必要です」
* 「宅建業者としての説明義務の観点から問題になり得ます」
* 「行政窓口に確認する材料になり得ます」
* 「一般論としては、費用の性質・根拠・任意性が明確に説明されることが重要です」
* 「第三者にも説明できる根拠資料の確認が重要です」
* 「外部に説明できない費用請求と評価される余地があります」

## 6. 重要なデータの流れ

本プロジェクトでは、以下の流れが重要である。

```text
入力フォーム（FlowRoot → PreContractFlow / PostContractFlow）
→ DiagnosisInput2
→ /api/diagnose-v2
→ diagnoseV2()
→ DiagnosisResult2
→ DiagnosisResultV2
→ 将来的な課金導線（980円想定）
→ 将来的なメール生成（Claude API想定）
```

課金導線・メール生成は将来的な接続先であり、2026年5月時点では未実装または未接続である。  
旧フロー（DiagnosisForm.tsx）に課金・メール生成関連の実装が残っている可能性があるため、現在の接続状態を確認せずに削除・統合・置換してはならない。

この接続を断ち切る変更は禁止。

UIだけを修正する場合でも、以下を必ず確認する。

* 入力された値が DiagnosisInput2 に入っているか
* DiagnosisInput2 が診断APIに渡っているか
* 診断結果が結果画面に表示されるか
* emailTone / fees / timeline / agentResponse が途中で失われていないか
* 課金後のメール生成に必要な emailStructure またはそれに相当する情報が結果に含まれているか、または将来接続可能な形で保持されているか

## 7. 重要ファイル・重要APIの扱い

### lib/diagnose_v2.ts（診断ロジックの核心）

現行V2診断ロジックの中核である。

変更禁止ではないが、意味のある変更を行う場合は、事前に以下を説明すること。

* どの Issue に影響するか
* severity が変わるか
* strategy が変わるか
* issue id が変わるか
* 結果表示に影響するか
* メール生成に影響するか
* 既存テストに影響するか
* 他の費目や契約段階に副作用がないか

**明示的な指示なしに severity / strategy / issue id / 出力構造を変更してはならない。**

### lib/types_v2.ts（型定義の核心）

現行V2型定義の中核である。

変更禁止ではないが、変更する場合は以下を確認すること。

* API との互換性
* 診断ロジックとの互換性
* 結果表示との互換性
* メール生成との互換性
* 既存データ構造との整合性

型を整理する場合でも、既存の値が失われる変更は禁止。

### app/api/diagnose-v2/route.ts（V2診断API）

DiagnosisInput2 を受け取り、diagnoseV2() に渡し、DiagnosisResult2 を返す中核接続である。

UI・型・診断ロジックを変更する場合でも、このAPIの入出力互換性を壊してはならない。

明示的な指示なしに、削除・統合・置換・レスポンス形式変更を行ってはならない。

### app/api/diagnose/route.ts（既存API）

既存APIである。

新APIやV2系の作業中でも、明示的な指示なしに変更しない。

既存フローとの互換性が残っている可能性があるため、削除・統合・置換は慎重に行うこと。

### components/DiagnosisForm.tsx（旧フロー・保護対象）

このファイルは旧フロー由来であり、現在の主表示ではない可能性がある。

しかし、以下が含まれている可能性がある。

* Stripe課金処理
* メール生成処理
* 既存API連携
* 新フローへ移植すべきロジック・文言・UI部品

**明示的な指示なしに、削除・大幅整理・未使用扱い・課金処理の削除・メール生成処理の削除を行ってはならない。**

DiagnosisForm.tsx は永久保存対象ではない。  
将来的には、旧フロー内の課金処理・メール生成処理・既存API連携を新フローへ安全に移植し、動作確認が完了した後に段階的に削除・整理する。

ただし、移植計画・確認項目・代替実装がない状態で削除してはならない。

### components/DiagnosisResult.tsx（既存結果表示）

既存結果表示に関わる重要ファイルである。

互換性を壊す変更は禁止。

表示改善を行う場合でも、診断結果の意味、Issue の扱い、severity / strategy の表示意図を変えてはならない。

### components/DiagnosisResultV2.tsx（新結果表示）

新結果表示に関わる重要ファイルである。

現行設計では、費目別の4バケット分類を維持すること。

```text
delete      → 赤系：不要の可能性が高い・削除を求める
free_rent   → オレンジ系：フリーレント転換を提案する
confirm / admin_check → 黄系：根拠確認が必要
record      → 緑系：払う根拠あり・記録固定
```

この分類を、明示的な指示なしに崩してはならない。

## 16. テスト方針

診断ロジックを変更する場合、既存テストがあれば必ず実行する。

テストが存在しない場合、診断ロジックを変更する前に、最低限の代表ケーステストを追加することを提案する。

優先テスト対象：

* 重説前支払い（payment_before_juusetsu）
* 仲介手数料の原則説明なし（agency_fee_principle_not_explained）
* 鍵交換代の実施証明なし（key_exchange_no_evidence）
* 任意費用の説明なし（optional_denied_if_refused）
* 保証会社のグループ会社未開示（guarantor_group_company_unexplained）
* 複数費目選択時の複合論点
* emailTone / fees / timeline / agentResponse が payload に残るか

テストは、単なるコード品質のためではなく、ユーザーが毎回スクリーンショットで全挙動を確認しなくても済むようにするための安全装置である。

## 17. UI修正の注意点

UI修正では、見た目だけでなく、ユーザーの理解プロセスを壊さないこと。

特に以下に注意する。

* なぜこの質問を聞いているのかが分かるか
* 同じような質問が重複していないか
* 入力順序が自然か
* 契約前・契約後・再契約などの段階が混乱しないか
* 費目選択後に、その費目固有の確認に自然につながるか
* スマホで読めるか
* 長文になりすぎていないか
* ユーザーが次に何をすればいいか分かるか

1. サービス思想を壊さない

7. そのうえでコードを整理する

「整理」「簡略化」「不要削除」は、収益化・メール生成・診断ロジック・旧フロー互換性を壊す可能性があるため、必ず慎重に行うこと。

* 「未使用」「不要」「移植済み」と判断したくなった場合

不確実な場合は、実装より先に影響範囲を説明すること。

## 20. 外部検証可能性の観点（重要）

このサービスの核心思想：

「外部で検証可能な費用を請求していたか」

業者が見せない・回答しない・はぐらかす・結論だけ言う行為は、
それ自体が記録になり、行政窓口・消費者センターへの
相談材料になる。

どちらに転んでもユーザーに有利：
- 業者がちゃんと説明する → 正当な費用かどうかわかる
- 業者が説明しない → それが記録になる

この観点を診断ロジック・結果表示・メール生成・
successページの全ての設計に反映させること。

## 21. 軸B：実態の不整合（追加）

軸Aの説明義務に加えて、以下の実態確認の軸を維持する。

費用の実態が存在するか：
- 鍵交換：シリンダーが新品か・鍵番号が一致するか・
  業者への支払い領収書があるか・流用でないか
- クリーニング：業者名・実施日・作業内容・
  業者への支払い領収書があるか

書類間の不整合：
- 重説・賃貸借契約書・請求書・領収書で
  費用の名目・金額・発行者が一致するか
- 日付の順序が適切か
  （重説→契約締結→役務実施→費用受領が正しい順序）
- 重説前の費用請求・受領がないか

仲介手数料の実態：
- 1ヶ月分請求の場合、貸主からも受領していないか
- 合算で1ヶ月分を超えていないか
- なぜ満額かの説明があったか

## 22. 特約・特記事項の扱い（重要）

「特約に書いてある」「読み上げた」は説明義務を
果たしたことにはならない。

本来貸主負担である費用を借主負担にするなら、
その根拠・理由の説明が必要。

任意費用であることの説明がなければ、
借主は何に同意したかわからない状態でサインしている。

特約の有効性の条件（判例・国交省ガイドライン）：
1. 特約の必要性があり暴利的でない
2. 借主が通常と異なる義務を負うことを認識していた
3. 借主が義務負担の意思表示をしていた

「特約があれば有効」という誤解に乗っかってはいけない。
「同意の実質的な有効性」を問う軸を常に維持すること。

「特約に書いてあります」という業者の回答は
根拠の説明になっていない。これも記録になる。

## 23. successページの設計原則

### 役割分担

診断段階：何が問題か・なぜ問題か
successページ：次に何をするか・何が起きるか・どう動くか

診断で伝えたことの繰り返しをしない。
冗長な重複を避ける。

### 契約前後の完全分岐

契約前（署名前）：
- 全ての選択肢が開いている
- 削除・フリーレント転換・他社比較・火災保険選択が可能
- 録音の活用（合法・一方当事者の同意で可）
- 季節・タイミング別の戦略を提示
- 繁忙期（1〜3月）：フリーレント転換が通りやすい
- 閑散期（4〜8月）：削除交渉・他社比較が有効
- 難易度：低

契約後・支払前：
- 支払いを保留しながら確認できる
- 根拠不明確な費用への支払い留保の方法を伝える
- 難易度：中

契約後・支払済み：
- 記録の固定が最優先
- 書類間不整合チェックリストを渡す
- 返金・減額より記録を積んで選択肢を広げる方針
- 行政窓口・消費者センターへの持ち込み方を伝える
- 難易度：高

### 業者の返答パターンと記録

以下の全パターンが「記録になる」という設計：

「問題ありません」→ 何が問題ないかの根拠を求める
「弊社の規定です」→ 規定の文書開示を求める
「一般的な費用です」→ 算定根拠を求める
「特約に書いてあります」→ 借主負担にする根拠を求める
返信なし・無視 → 送信記録を保存

### UIの情報設計原則

1. 今すぐやることを1つだけ最初に出す
2. 追加情報は折りたたみで出す
3. 契約前後で表示するブロックを完全に変える
4. チェックリストは1画面1項目
5. 診断で伝えた内容を繰り返さない

### 仲介手数料・火災保険の追加情報

仲介手数料：
- 0ヶ月分の業者も存在する（オーナーから受領）
- 他社比較が有効な交渉材料になる
- 「他社で0.5ヶ月の見積もりが出ています」という
  事実が交渉材料になる

火災保険：
- 加入義務はあるが保険会社・プランは借主が選べる
- 貸主が指定する最低補償内容を満たせば他社プランでよい
- 相場より安くなる可能性がある

## 25. 契約前と契約後は別サービスとして設計する（最重要）

このサービスは「契約前（見積もり段階）」と
「契約後（支払済み・トラブル）」で
本質的に異なる2つのサービスである。

### 契約前（pre_contract）
- 手元にあるのは見積書のみ（契約書・重説はまだ）
- 目的：費用を外す・下げる・他社比較する
- トーン：前向き・条件が整えば契約したい
- 質問：各費目1〜2問・「説明されたか」中心
- 軸：外せるか・下げられるか（交渉）

### 契約後（post_contract）
- すでに支払い済み・実態が存在する
- 目的：根拠確認・記録・行政相談
- トーン：すでに払った前提・記録固定
- 質問：実態・時系列・書面不整合（詳細）
- 軸：外部検証可能性・説明義務

### 実装方針（整合性を守る）
- 既存の detectFeeIssues は「契約後用」として一切変更しない
- 契約前は detectPreContractIssues を新設して分岐
- 型（Issue2・FeeEntry・DiagnosisResult2）は共有
- メール生成・結果表示・課金処理は共有（timingで分岐）
- PreContractFlow / PostContractFlow は既に分離済み

## 26. 契約前フローの確定設計（実務反映版）

### 基本思想

契約前は「説明されたか」を聞かない。
重要事項説明は契約直前に行われるため、
契約前段階では「説明されていない」のが当然。

代わりに「交渉できる状況か」を実務軸で把握する。
質問はベクトルを変える：説明軸 → 実務・交渉軸。

各質問は意義を明示し、ユーザーが
「自分のことを正確に答えている」と感じる設計にする。

### 申込状況による交渉力の違い

- まだ申込前：「他社で検討」と自然に言える・最強
- 申込済み・審査待ち：業者は逃したくない・交渉余地大
- 審査通過済み・見積もり受領：契約直前・既に決まっている状況

### 聞くこと（5問＋費目）

Q1：家賃（必須）
　意義：相場レンジの基準として使います

Q2：契約予定時期（必須）
　意義：繁忙期と閑散期で交渉戦略が変わるため
　選択肢：1〜3月（繁忙期）/ 4〜8月（閑散期）/ 9〜12月（通常期）

Q3：物件への申込状況（必須）
　意義：申込のタイミングで交渉力が変わるため
　選択肢：
　- まだ申込んでいない
　- 申込済み・審査待ち
　- 審査通過済み・見積もり受領済み

Q4：他社で同じ条件を比較していますか？（任意）
　意義：他社比較は交渉材料になるため
　選択肢：している / していない / 比較を予定している

Q5：費目と金額（必須）
　各費目について：何の費目か・いくらか

### 診断結果の出し方（契約後との区別）

契約前の結果画面：
- メイン表示：「あなたの見積もり vs 一般的レンジ」
- 副表示：「調整余地のある費目（金額付き）」
- 戦略：「時期＋申込状況」から最適な交渉戦略を提示
- トーン：前向き・チャンス・解決前提
- 色：青〜緑系（前向き）

契約後の結果画面：
- メイン表示：4バケット（論点・重大・注意）
- 副表示：論点リスト
- 戦略：根拠確認・記録・行政相談
- トーン：記録・確認
- 色：赤・オレンジ・黄（警告色寄り）

### 質問の見せ方

- 各質問の上または下に「（〇〇のため）」と意義を表示
- 回答後にミニフィードバック
  例：「閑散期・申込前ですね。
        削除交渉が通りやすい状況です」
- 「自分のことを正確に答えている」感を出す

### severity・strategy の方針

費目分類による strategy（基本）：
- 任意費目（消毒・サポート・書類作成費）→ strategy: delete
- 貸主負担費目（鍵交換・クリーニング）→ strategy: free_rent
- 制度費目（仲介・保証・火災）→ strategy: confirm または free_rent
- 慣行費目（礼金）→ strategy: free_rent

severity は「申込状況」で調整：
- 申込前 → 強気で出せる（high）
- 申込後・審査待ち → medium
- 審査通過済み → medium（既に進んでいるため慎重に）

### レイヤー1（無料の相場診断）への布石

家賃と時期だけで相場レンジを出せる構造にしておく。
将来、無料診断として独立できるようにする。
（実装は当面後回し・セクション31として記録）

### 旧設計からの変更点

- 「説明されたか」系の質問を全廃
- 契約前フローのPreContractDetailを実務寄りに再設計
- detectPreContractIssues も新ベクトルに合わせて書き直し
- これらの修正は次の実装フェーズで実施
